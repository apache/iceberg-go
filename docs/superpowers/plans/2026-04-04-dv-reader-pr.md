**Architecture:** The DV reader pipeline has 3 layers: (1) `RoaringPositionBitmap` — 64-bit position bitmap backed by `RoaringBitmap/roaring/v2`, (2) DV serialization — envelope format with length/magic/bitmap/CRC, (3) Scanner wiring — `readAllDeleteFiles` reads DVs from puffin files, `recordsFromTask` merges DV positions into the existing positional delete pipeline. All three layers exist on the `dv-reader` branch. This plan fixes bugs, adds missing tests, and removes writer code (deferred to PR2).

**Tech Stack:** Go 1.25, `RoaringBitmap/roaring/v2`, `apache/arrow-go/v18`, `apache/iceberg-go/puffin`

```go
// serializeDVForTest is a test-only helper that serializes a bitmap
// into the DV binary format (length + magic + bitmap + CRC-32).
// Production writer will be added in a follow-up PR.
func serializeDVForTest(t *testing.T, bitmap *RoaringPositionBitmap) []byte {
	t.Helper()
	bitmap.RunOptimize()

	var bitmapBuf bytes.Buffer
	require.NoError(t, bitmap.Serialize(&bitmapBuf))
	bitmapBytes := bitmapBuf.Bytes()

	bitmapDataLength := dvMagicSize + len(bitmapBytes)
	totalSize := dvLengthSize + bitmapDataLength + dvCRCSize
	out := make([]byte, totalSize)

	binary.BigEndian.PutUint32(out[0:dvLengthSize], uint32(bitmapDataLength))
	binary.LittleEndian.PutUint32(out[dvLengthSize:dvLengthSize+dvMagicSize], DVMagicNumber)
	copy(out[dvLengthSize+dvMagicSize:], bitmapBytes)

	crc := crc32.ChecksumIEEE(out[dvLengthSize : totalSize-dvCRCSize])
	binary.BigEndian.PutUint32(out[totalSize-dvCRCSize:], crc)

	return out
}
```

### Task 3: Fix race condition in readAllDeleteFiles DV path

The DV errgroup in `arrow_scanner.go:124-167` has a race: `err = g.Wait()` runs inside a goroutine that closes `dvChan`, but the outer code reads `err` after `range dvChan` finishes. If the goroutine hasn't assigned `err` yet when the range completes, the error is lost.

**Files:**
- Modify: `table/arrow_scanner.go:124-167`

- [ ] **Step 1: Understand the race**

Current code (simplified):

```go
go func() {
    defer close(dvChan)    // channel closes
    for _, v := range uniqueDVDeletes {
        g.Go(func() error { ... })
    }
    err = g.Wait()         // err assigned AFTER close triggers range exit
}()

for dv := range dvChan { ... }  // drains, then falls through

if err != nil {            // may read stale err (nil) if goroutine hasn't run g.Wait() yet
    return nil, nil, err
}
```

The fix: capture the error from `g.Wait()` in a way that's guaranteed to be visible after the channel range completes. Since `close(dvChan)` happens after `g.Wait()` returns (deferred close runs last), the assignment `err = g.Wait()` actually IS sequenced before the close. The `defer close(dvChan)` executes after `g.Wait()` returns, so the range will exit after `err` is assigned.

**Wait — re-analysis:** The `defer close(dvChan)` runs when the goroutine's anonymous function returns. The function body is: (1) launch g.Go workers, (2) `err = g.Wait()`, (3) return — then defer fires `close(dvChan)`. So `err` IS assigned before `close`. The `range dvChan` blocks until close, then exits. Then `err` is read. This is actually safe because the goroutine writes `err` before closing the channel, and the main goroutine reads `err` after the channel is drained (which happens after close).

**However**, the `err` variable is shared with the parquet section above. If both sections run and the parquet section already set `err`, the DV section overwrites it. And vice versa. This is a correctness issue: if parquet deletes succeed but DV deletes fail, the parquet `err` (nil) gets overwritten by the DV `err` (error). But if DV deletes succeed and parquet fails, the DV `err = g.Wait()` (nil) overwrites the parquet error.

The real fix: use separate error variables for each section.
